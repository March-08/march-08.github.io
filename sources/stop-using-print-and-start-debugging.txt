**Introduction**\nI once read that in programming an error in code is called *bug *for a very specific reason. When computers were still huge mainframes (a long, long time ago) it happened that a bug got stuck in the gears and that’s why the program wouldn’t run anymore!\nToday by bugs we mean a very different thing. Everyone, even the most experienced and paid programmers in the world write code that contains bugs, but the skill lies in finding them. In this article, we find out how!\n## **Print and Jupyter Notebooks**\nIt is common for data scientists to write code using Jupyter Notebooks or Google Colab. I too find that they are super convenient to use, unfortunately, though debugging is not as convenient.\nIn these notebooks, we can split our code into cells, that way at the end of the execution of a cell, we can print out all the values of the variables we are interested in, and see if anything has gone wrong. Or at least we can get an idea of what’s going on inside the code.\nEven in cases where we do not use notebooks, we often use print() to figure out the value of variables and thus understand why the code behaved a certain way and fix it.\nFor example, suppose we want to execute the following code.\n```plain text\nadmin = get_user_input() #returns user input\n\nif admin == 'admin':\n  print(f'welcome {admin}')\n```\nWith the previous code, we know we have the variable admin. When this variable takes the value *‘admin’* we want to greet our admin.\nWe find, however, that this condition never occurs, even when it should occur, since it is our admin who is using the program.\nThen what novice programmers generally do is go and print out the admin variable and figure out what’s wrong.\n```plain text\nadmin = get_user_input() #returns user input\nprint('The value of admin is : ' , admin)\n\nif admin == 'admin':\n  print(f'welcome {admin} ')\n```\nWe note that the result of this print is as follows.\n```plain text\nThe value of admin is : admin\\n\n```\nAnd we find that our *get_user_input *function returns one* \\\\n* extra and that is why the condition never occurs. We are then ready to fix our code and run everything.** Bug found**!\nBut is this the best way to find the bug? Not really. In such simple cases where we have 4 lines of code maybe it is okay, but in more complex projects you have to prefer the debugger tool that is provided by various IDEs.\n## **Debug with VSCode**\n**IDE** stands for Integrated Development Environment and is a tool that allows you to write code by providing very useful tools such as a debugger.\nIn this article, we will look at the** VSCode debugger **but they all work pretty much the same, so even if you use PyCharm or any other IDE don’t worry.\nLet’s start by writing a simple code in Python in VSCode. The example code is as follows.\n```plain text\nfrom random import random, randint\nimport time\n\nclass User:\n    def __init__(self, name , psw) -> None:\n        self.name = name\n        self.psw = psw\n\n    def say_hello(self):\n        print(f'Hello {self.name}')\n        self.get_name_len()\n\n    def get_name_len(self):\n        print(len(self.name))\n\n    def __str__(self) -> str:\n        return f\"{self.name} , {self.psw}\"\n\n\nif __name__ == '__main__':\n\n    for _  in range(3):\n        i = int(randint(0,10))\n\n        name = f'user_ {i}'\n        psw = f'psw_ {i}'\n        user = User(name, psw)\n        user.say_hello()\n```\nIn this code, we created a User class, where each user is described by name and password.\nPlus a user has two methods, such as *say_hello *which greets the user, *get_name_len *which returns the length of the user’s name, and the *__str__ *method which defines how a user should be printed.\nFinally in the *main *we go to create three random users and have them use the *say_hello *method.\nLet’s now go to introduce an error. For example in the *get_name_len *method, we only print the length of *self* instead of *self.name*.\n```plain text\nfrom random import random, randint\nimport time\n\nclass User:\n    def __init__(self, name , psw) -> None:\n        self.name = name\n        self.psw = psw\n\n    def say_hello(self):\n        print(f'Hello {self.name}')\n        self.get_name_len()\n\n    def get_name_len(self):\n        print(len(self))\n\n    def __str__(self) -> str:\n        return f\"{self.name} , {self.psw}\"\n\nif __name__ == '__main__':\n\n    for _  in range(3):\n        i = int(randint(0,10))\n\n        name = f'user_ {i}'\n        psw = f'psw_ {i}'\n        user = User(name, psw)\n        user.say_hello()\n```\nNow if we go to execute the code we will get the following error.\n![](https://miro.medium.com/v2/resize:fit:570/1*nMRGPSPLTgbAhneQbkMJTQ.png)\nError (Image By Author)\nOur purpose now is to try to find the error using the VSCode debugger.\nThe first thing to do is to insert a ***Breakpoint ***at a certain point in your code. With the Breakpoint, you tell your compiler to **pause the run **at that point so that you can inspect the various variables in your code, and then continue the run only when you want it to.\nTo enter a Breakpoint click to the far left of your code and a red dot will appear.\n![](https://miro.medium.com/v2/resize:fit:700/1*Ua7McZbBe0PARLaqTunzLw.png)\nBreakpoint (Image By Author)\nNow let’s run the code in debug mode by clicking on the following icon.\n![](https://miro.medium.com/v2/resize:fit:505/1*ijKmp9eBNJ31Vxh1zJQ9ng.png)\nDebug (Image By Author)\nOnce you click you will see that your code will start the run but stop almost immediately.\nYou should now have a screen like this.\n![](https://miro.medium.com/v2/resize:fit:700/1*cLFYmIBUIo194TTHFDb0iQ.png)\nDebug (Image By Author)\nSo, the yellow line with that symbol next to it, tells you where the code stopped, right where you put the Breakpoint.\nIn the upper left corner, on the other hand, you can see the contents of your variables within the code. Many of these variables seem meaningless since they are pointers to functions you have in the code. But if you see well there should be a User variable since before the Breakpoint we have already created a user. In my case, I can see the content which is as follows.\n![](https://miro.medium.com/v2/resize:fit:382/1*7cUrVIz6GpDNfVB8NgOfiA.png)\nVariables (Image By Author)\nThis way I know what the content of my user variable is and I have already saved myself from writing a print().\nNow we can start using the main debugger commands found at the top of the bar made like this.\n![](https://miro.medium.com/v2/resize:fit:593/1*eNr5X-t6muHj_DaB4FOoSg.png)\nCommands (Image By Author)\nThe main commands are those highlighted in red. The first allows us to go forward one line of code to see what happens. The second allows us to go inside a function and see what happens. And the last one allows us to go outside the scope of a function.\nSince our program is now stuck on the *user.say_hello()* function let's click on the down arrow to see what happens inside this function.\nOnce clicked you will see that the execution will continue within this function.\n![](https://miro.medium.com/v2/resize:fit:529/1*I6ZW2HMh1EzcjI6GBmocrw.png)\nsay_hello (Image By Author)\nNow we need to understand what the stack is for. The stack tells you basically how many functions you are in. In fact, if you see in the lower left-hand corner now a line has been added.\n![](https://miro.medium.com/v2/resize:fit:406/1*wzK6g85So-FyACPKy4Hkqw.png)\nStack (Image By Author)\nThis tells us that we are now in the scope of the *say_hello* function so if we exit this function by clicking the up arrow, we will be back in the main.\nNow let’s move forward one line of code with the *forward *command.\n![](https://miro.medium.com/v2/resize:fit:700/1*d7haePAQvFJBL6MJZRicbA.png)\nForward Command (Image By Author)\nNow that we are over another function, we click the down arrow to go inside the function itself.\n![](https://miro.medium.com/v2/resize:fit:700/1*SMle4l-7GqKFZKYWBbxDiA.png)\nStack (Image By Author)\nYou see that because we have entered inside another function the stack has increased, so we are in a function inside a function (no, this isn’t Inception).\nObviously, we can enter more than one unique Breakpoint. So let’s stop the debug and relaunch it by entering 2 Breakpoints this time.\n![](https://miro.medium.com/v2/resize:fit:675/1*7w5QN5h2Djzzm_CWYEjVSQ.png)\nTwo Breakpoints (Image By Author)\nSo in this case by clicking on *forward*, we are going to stop at the second Breakpoint.\nIf we now click forward again, the code will go to stop, but this time because **we found the error**!\n![](https://miro.medium.com/v2/resize:fit:700/1*egz__m_xKf0qQ9m4EaEOwA.png)\nError Found (Image By Author)\nWe succeeded in our intent. Error found, and we did not dirty our code with useless prints!\nOnly one thing remains to be said. What is the watch section for? Well, to look at variables.\nSuppose we save all our users in a list.\n```plain text\nfrom random import random, randint\nimport time\n\nclass User:\n    def __init__(self, name , psw) -> None:\n        self.name = name\n        self.psw = psw\n\n    def say_hello(self):\n        print(f'Hello {self.name}')\n        self.get_name_len()\n\n    def get_name_len(self):\n        print('ciao')\n        print(len(self.name))\n\n    def __str__(self) -> str:\n        return f\"{self.name} , {self.psw}\"\n\n\nif __name__ == '__main__':\n\n    users = []\n    for _  in range(3):\n        i = int(randint(0,10))\n\n        name = f'user_ {i}'\n        psw = f'psw_ {i}'\n        user = User(name, psw)\n        user.say_hello()\n        users.append(user)\n\n    print('end of this tutorial')\n```\nNow we want to see what happens to this list as we go through the code.\nSo let’s click on the plus button and add the viable users under watch. In fact, let’s add *len(users) *as well.\n![](https://miro.medium.com/v2/resize:fit:379/1*jsge0fn6yPVq5GhAyAJhFw.png)\nWatch (Image By Author)\nNow as you move forward with your code with the forward command, you will see the list fills up and you will be able to inspect the objects within it.\n![](https://miro.medium.com/v2/resize:fit:430/1*Cuw7HwK-wxDCVp1wUjNMhg.png)\nAfter some iterations (Image By Author)\n## **Final Thoughts**\nWell, these are the basics of using the debugger. It is a very useful and easy tool to use. Of course like all things, you have to get used to it.
