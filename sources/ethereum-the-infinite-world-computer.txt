to the state.\nEthereum purpose is not to be a currency, but ether are intended to be a *utility currency *to pay for the use of the world computer. \nMany similarities with bitcoin’s blockchain, but the big innovation here is that Ethereum’s blockchain can be programmed, so can support any kind of application. \n<empty-block/>\n- Ethereum’s nodes, instead of only tracking the state of the currency ownership, they track general-purpose data that can expressed in the format of *key-value tuple.*\n- Ethereum has a memory that store both code and data. \n- Ethereum can load code into it’s state machine and run the code, storing the results (state changes) into the blockchain → state changes are governed by consensus rules. \n- Ethereum code (smart contracts) unlike Bitcoin, is Turing Complete, it can compute any algorithm computed by any Turing Machine. \n \n<embed It can be divided into smaller units, the smallest possible unit is called ***wei.***\n<empty-block/>\n<empty-block/>\n$$\n1 ETH = 10^{18}wei\n$$\nIn order to interact with the Ethereum ecosystem you need a **wallet.**\nA wallet is a software that helps you manage your account by holding your keys and broadcasting your transactions.\nSome of my favorite wallets are:\n- [Metamask](https://metamask.io/en-GB) 🐺\n- [Rabbit](https://rabby.io/) 🐇\nYou can install both as your browser extension. \nWhen you’ll create an account using for example MetaMask, it will ask you to generate a ***mnemonic backup ***consisting of 12 english words that you can use to recover access to your founds or to switch to another wallet. \n<empty-block/>\nIn MetaMask you will have access to many different networks, among which some test network. Testnets are useful for learning of developing purposes, because every currency spent there has no real value.\nSo switch to the **Sepolia **test network in your MetaMask wallet (in a dropdown menu). \n<empty-block/>\nTo load your wallet with some ETH, you can go on a [faucet website](https://cloud.google.com/application/web3/faucet/ethereum/sepolia), insert your wallet address and request some funds. \nThe Faucet website will elaborate a transaction to send you funds. All transaction are visible on a block explorer. The most famous one is [Etherscan](https://etherscan.io/), while for Seoplia you can use [this one](https://sepolia.etherscan.io/address/0x3e8017C24352CBd3130e728Fe2FE7332BCF9ED58).\nEvery transactions requires the payment of a fee, called **gas**, used to reward miners to validate the transaction. \nThe gas price is not stable changes every time depending on the network congestion.\nE.g:\n- Gas price 3 gwei ( $`10^9`$ wei)\n- A simple transaction requires 21000 gas units\n- Total fee = 3\\*21000 = 63000 gwei\n<empty-block/>\nEther is not merely a currency, it is used to pay for running ***smart contracts, ***which are programs that run on an emulated computer called ***Ethereum Virtual Machine (EVM)***.\nEach node of the Ethereum network, has a local copy of the EVM to validate contracts execution, while the Ethereum blockchain records the changing state of this world computer as it processes smart contracts. \n## Wallets and Contracts\nWallet such as MetaMask that hold private keys, are called ***Externally Owned Account (EOA)***.\nAnother type of account exists called ***contract account. ***This account has a smart contract code which has some logic.\nContracts can send and receive funds like EOAs. But when a transaction (tx) destination is a contract address, it causes the contract to run in the EVM, using the tx and the tx data as input. Tx contain indeed data specifying which function of the contract to *call *(run). <br>\nSince contracts don’t have private keys, they cannot *initiate* a tx. Only an EOA can initiate a tx.\n<empty-block/>\n## Simple Smart Contract\nEthereum has [different high level languages](https://ethereum.org/developers/docs/smart-contracts/languages/) to write contracts, the most common one is Solidity. \nA contract code is then converted into EVM bytecode and executed.\n<empty-block/>\nLet’s write a faucet contract. It will have a function to withdraw some money (faucet sends money to who requests), and a function to receive money (users send money to contract).\nThe contract is commented to be understood. \n```solidity\n// SPDX-License-Identifier: CC-BY-SA-4.0\n\n// Version of Solidity compiler this program was written for\npragma solidity 0.8.30;\n\n// this line specifyis that we are writing a contract. Like class in OOP languages\ncontract Faucet {\n\n    // A particular (defualt) function used to accept payments by anyone\n    // called fallback or default function\n    // is called it users triggers this contract without specifying any particular function\n    receive() external payable {}\n\n    // function users can use to withdraw some money\n    function withdraw(uint withdraw_amount) public {\n        \n        // Make a check to lLimit withdrawal amount\n        require(withdraw_amount <= 100000000000000000);\n\n        // Send the amount to the address that requested it\n        payable(msg.sender).transfer(withdraw_amount);\n    }\n}\n```\nThe ***msg*** object is one of the inputs that all conctacts can access. It represents the transaction that triggered the execution of this contract.\nWe cast the sender of the transaction as payable so that we can transfer money to him/her →  `payable(msg.sender).transfer(withdraw_amount)`\n \nThere are some IDEs that come with a Solidity compiler, the most popular one being [***Remix***](https://remix.ethereum.org/).\nIn Remix you can navigate into the contracts folder and create a *Faucet.sol* Solidity file, and paste the contract.\nThen on the compiler icon you can compile the code. In the *ENVIRONMENT* dropdown menu, you can choose “W*alletConnection”, * so it will automatically select the chain selected in your wallet → Sepolia.\nIn the *Account* box, you’ll see your wallet address.\n<empty-block/>\nAfter the contracts it’s compiled you can *deploy *** **by clicking on the button and by signing a tx. Transactions\n> Transactions are signed messages originated by an externally owned account, transmitted by the Ethereum network, and recorded on the Ethereum blockchain.\nTransactions are the only From the yellow paper:\n<callout icon=\"💡\" color=\"gray_bg\">\n\tnonce: A scalar value equal to the number of transactions sent from this address or, in the case of accounts with associated code, the number of contract-creations made by this accout.\n</callout>\n<empty-block/>\nSome scenarios example when a nonce is needed:\n1. You send 2 transactions. You just realised after sending the transactions, that you don’t have enough ether for both, so you would like the protocol to reject the second one because obviously you prioritise the first one. But the network doesn’t know which one is the first or second one because the transactions come in a “random” order to the nodes. Here a nonce solves the situation. \n2. Imagine you send a transaction of 10 eth to buy your knew playstation-5. What if some node replays the transaction by resending it multiple times to the network? He could do this and you will spend 10 eth over and over again. But if in the transaction there is a nonce we know that a particular transaction has already been included in the chain, and new transactions will need new nonces which will affect the transaction hash.  \n<empty-block/>\nThe Ethereum network processes transactions sequentially, meaning that if you send a tx with nonce 0 and then a tx with nonce 2, the second transaction will not be included in any block but it will wait in a ***mempool*** until a tx with nonce 1 is received. \n<empty-block/>\nThe <span color=\"green\">gas price </span>is the price that the transaction originator is willing to pay for a gas unit. Different kind of transaction require a different amount of gas unit (e. g a simple EOA to EOA tx requires 21k units). Gas price can change based on the state of the network, so it can be only estimated averaging the gar price of the latest blocks. \n \nThe <span color=\"purple\">gas limit </span>instead,  is the maximum number of gas units a transaction originator is willing to buy in order to complete the transaction. Oracles provide a trustless (or near-trustless) way of getting extrinsic data (in the real world), such as the results of football games, on chain. They bridge the gap between the off-chain world and the smart contracts.\nSome data that might be provided by an oracle could be: random numbers, exchange rate, capital markets, weather data, sporting event etc.\nThe common design pattern of an oracle is the following:\n- collect data from an off-chain source\n- transfer data on-chain with a signed message\n- make the data available by putting it in a smart contract’s storage\nOnce data is in a smart contract it can be retrieved by others.\nThere are mainly 3 types of oracles:\n1. ***immediate-read: ***when the data is needed for an immediate decision: “is this person over 18 yo?”. The lookup is done when the info is needed and never again. This type of oracle store data once in its contract storage. A university might set up an oracle for the certificates of academic achievement of past students. \n2. ***publish-subscribe: ***oracle that provide broadcast services for data that is expected to change. When the oracle is updated with new information it flags signals to those who are considered “subscribed”. Interested parties should listen for updates. Examples are: price feeds, weather info, social statistic, traffic data etc.\n3. ***request-response: ***this is where the data space is too huge to be stored in a smart contract, and users only need a small part of it. Such systems have off-chain infrastructure. In this pattern usually an EOA transact with a decentralised application, resulting in an interaction with a function defined in the oracle smart contract. This function initiates the request to the oracle, with the associated arguments detailing the data requested. Once the tx has been validated, the oracle request can be observed as an EVM event emitted by the oracle contract, or as a state change; the arguments can be retreived and used to perform the actual query of the off-chain data source.  Finally, the resulting data is signed by the oracle owner, attesting to the validity of the data at a given time, and delivered in a transaction to the decentralised application that made the request. \n<empty-block/>\nHow we can trust off-chain methods? There is the possibility that data as been tampered with in transit.\nTwo common approaches to validate data integrity are: ***authenticity proofs*** and ***trusted execution environments (TEE)***\nAuthenticity proofs, are cryptographic guarantees that data has not been tampered with, By verifying the authenticity proof on-chain, smart contracts are able to verify the integrity of the data before operating upon it. \n<empty-block/>\nTEE oracles instead (such as Town Crier), utilize hardware-based secure enclaves to ensure data integrity. \n<video src=\"https://www.youtube.com/watch?v=MREwcSo0uz4\"></video>\n \nOracles can bes used also to perform computation, not only data storage. \nYou can trust a centralised but auditable service like Oraclize, that provide a service for decentralised applications that requested a particular computation.\nOtherwise you can use a more decentralised alternative like [TrueBit](https://truebit.io/).
