Only in this way can we understand how it will scale. A sorting algorithm might seem optimal on a toy case where we have to sort an array of length 3, but be highly inefficient when we have an array of length 10M.\nFor this reason, it is not trivial to compare the performance of two different algorithms. One might seem faster in specific instances, and the other faster in others; we need a mathematical and **formal method to properly evaluate** them.\nAs we mentioned, an algorithm might behave differently depending on the input instance on which it works. But then in which instances is it right to monitor its performance? On the ones on which it is fastest? Or slower?\nUsually, the **worst-case running time** is used. That way we are sure what will happen in the worst possible case. And if the worst-case is not that bad we are happy 😊.\nAnother method is to rely on the **average-case**. That is, we randomly create input instances, and average the running times. With this method, however, we have a fundamental problem: How do we choose those random instances?\nDue to these problems, we prefer to rely on the worst-case scenario.\n## **What does efficiency mean?**\nBut if **we don’t have a benchmark**, how do we know how efficient an algorithm we develop is?\nWe can start with something simple. For every problem, we almost always have a **naive algorithm** that solves it,** brute-force**. This consists of searching the entire possible search space in order to find a solution. In short,** we try them all**! Obviously, this **is very inefficient** because the search space can be huge.\nThen we must try to formalize efficiency on a mathematical plane on which we can agree. Let’s say **an algorithm is efficient if it has a polynomial running time**.\nHence if there exist two constants c, d \\> 0 such that on each input instance of length N the running time can be **upper bounded by cN\\^d computational steps** (which may correspond, for example, to assembly code instructions).\nThus the running time remains proportionally linear to the size of the input instance.\n# **Asymptotic Notation**\nIn Computer Science when we express the running time of an algorithm by a function we 𝙙𝙤 𝙣𝙤𝙩 𝙘𝙤𝙣𝙨𝙞𝙙𝙚𝙧 𝙩𝙝𝙚 𝙘𝙤𝙣𝙨𝙩𝙖𝙣𝙩𝙨 𝙖𝙣𝙙 𝙩𝙚𝙧𝙢𝙨 𝙤𝙛 𝙡𝙤𝙬𝙚𝙧 𝙤𝙧𝙙𝙚𝙧 𝙩𝙚𝙧𝙢𝙨. Therefore let’s look at some of them so that we can recognize patterns and know how to distinguish them when we find them in our work.\n![](https://miro.medium.com/v2/resize:fit:1400/1*U4f4--IXjfjZ0qZNZPzzUw.png)\nsrc: [https://www.geeksforgeeks.org/analysis-algorithms-big-o-analysis/](https://www.geeksforgeeks.org/analysis-algorithms-big-o-analysis/) (CC BY-SA)\n## **Linear Time**\nAlgorithms of this type, **have a bound of O(n)**. This means that for each item in the input (imagine the input as a list of n items) they need to process a constant number of operations. For example in the problem of finding the maximum of a list, we only need to look once at all the items in the list and save the largest one from time to time, thus requiring n steps.\n```plain text\ndef find_maximum(arr):\n    max_val = arr[0]  # Initialize max_val with the first element of the array\n\n    # Iterate through the array to find the maximum value\n    for num in arr[1:]:  # Start from the second element since we initialized max_val with the first\n        if num > max_val:\n            max_val = num  # Update max_val if a larger value is found\n    return max_val\n\n# Example usage:\narray = [12, 45, 78, 23, 56, 91, 34, 67]\nmaximum = find_maximum(array)\nprint(f\"The maximum value in the array is: {maximum}\")\n```\nAnother example of an algorithm in linear time is one that performs the merge of two ordered lists while maintaining the order.\nHere you can read the pseudocode of the algorithm:\n```plain text\nMergeLists(list1, list2):\n    Initialize an empty list, mergedList, to store the merged result\n    Initialize pointers i, j for list1 and list2, respectively, starting at index 0\n\n    while i < length of list1 and j < length of list2:\n        if list1[i] <= list2[j]:\n            Append list1[i] to mergedList\n            Increment i by 1\n        else:\n            Append list2[j] to mergedList\n            Increment j by 1\n\n    // If elements are remaining in list1 or list2, append them to mergedList\n    while i < length of list1:\n        Append list1[i] to mergedList\n        Increment i by 1\n\n    while j < length of list2:\n        Append list2[j] to mergedList\n        Increment j by 1\n\n    return mergedList\n```\nLet’s look at an implementation in Python now of this algorithm.\n```plain text\ndef merge_lists(list1, list2):\n    merged_list = []  # Initialize an empty list to store the merged result\n    i = j = 0  # Initialize pointers for list1 and list2, respectively\n\n    while i < len(list1) and j < len(list2):\n        if list1[i] <= list2[j]:\n            merged_list.append(list1[i])  # Append list1[i] to mergedList\n            i += 1  # Increment i by 1\n        else:\n            merged_list.append(list2[j])  # Append list2[j] to mergedList\n            j += 1  # Increment j by 1\n\n    # If elements remain in list1 or list2, append them to mergedList\n    while i < len(list1):\n        merged_list.append(list1[i])\n        i += 1\n\n    while j < len(list2):\n        merged_list.append(list2[j])\n        j += 1\n\n    return merged_list\n\n# Example usage:\nlist1 = [1, 3, 5, 7]\nlist2 = [2, 4, 6, 8]\nresult = merge_lists(list1, list2)\nprint(\"Merged List:\", result)\n```\n## **O(nlog(n)) Time**\nThis is a very **common bound in computer science**. Because it’s the t**ime it takes to create recursive splits of the input n**, working on the smaller splits created, **and recombining the whole thing**.\nA classic example of this is **mergesort**, in which we divide the array into equal parts recursively, and each time we sort two pieces at a time to recombine everything.\n![](https://miro.medium.com/v2/resize:fit:1400/1*FPcHV6t8vX_FlRds4PyNBA.png)\nImage by Author\nHere is how to implement mergesort in Python.\n```plain text\ndef merge_sort(arr):\n    if len(arr) > 1:\n        mid = len(arr) // 2\n        left_half = arr[:mid]\n        right_half = arr[mid:]\n\n        merge_sort(left_half)  # Sorting the left half\n        merge_sort(right_half)  # Sorting the right half\n\n        merge(arr, left_half, right_half)  # Merging the sorted halves\n\ndef merge(arr, left_half, right_half):\n    i = j = k = 0  # Initialize pointers for left_half, right_half, and arr\n\n    while i < len(left_half) and j < len(right_half):\n        if left_half[i] < right_half[j]:\n            arr[k] = left_half[i]\n            i += 1\n        else:\n            arr[k] = right_half[j]\n            j += 1\n        k += 1\n\n    # Checking for any remaining elements in left_half or right_half\n    while i < len(left_half):\n        arr[k] = left_half[i]\n        i += 1\n        k += 1\n\n    while j < len(right_half):\n        arr[k] = right_half[j]\n        j += 1\n        k += 1\n\n# Example usage:\nmy_array = [12, 3, 67, 45, 9, 21, 34, 6]\nprint(\"Original array:\", my_array)\n\nmerge_sort(my_array)\nprint(\"Sorted array:\", my_array)\n```\n**Many algorithms have a computational cost of O(n\\*log(n)) because often the greatest cost is just in sorting an array**.\n## **Quadratic Time**\nSuppose the problem in which **we have n points** on the Cartesian plane. Each point is represented by coordinates (x,y). I want to **figure out what is the shortest distance between two points**. A simple algorithm could **take all pairs of points, calculate their distance**, and take the pair with minimum distance. But how long does it take?\nWell to take all the pairs, we can use combinatorial calculus, and this has a **cost of O(n²)**. Instead, we can calculate the distance between two points in a constant way using a simple formula. So the total cost of the algorithm is still quadratic.\nMore pragmatically, we can say that we encounter **quadratic cost algorithms when we have two nested for loops**. In which we scan a list twice:\n```plain text\nFor each input point(x_i, y_i):\n  For each other input point (x_i, y_i):\n    Compute distance d\n```\nIn Python it would be:\n```plain text\nimport math\n\ndef calculate_distance(point1, point2):\n    # Calculate Euclidean distance between two points\n    return math.sqrt((point2[0] - point1[0])**2 + (point2[1] - point1[1])**2)\n\ndef min_distance(points):\n    min_dist = float('inf')  # Initialize with positive infinity as an initial minimum distance\n\n    # Iterate through each point and calculate distances\n    for i in range(len(points)):\n        for j in range(i + 1, len(points)):\n            dist = calculate_distance(points[i], points[j])  # Calculate distance between points[i] and points[j]\n            if dist < min_dist:  # Update minimum distance if a smaller distance is found\n                min_dist = dist\n\n    return min_dist\n\n# Example usage:\ninput_points = [(1, 2), (4, 6), (7, 8), (3, 5), (9, 1)]\nresult = min_distance(input_points)\nprint(\"Minimum distance between two points:\", result)\n\n```\n# **Final Thoughts**\nKnowing how to **calculate the time** (and even space)** complexity** of an algorithm is very important. **If an algorithm is poorly thought out, even the best programmer in the world will not bring the code so efficiently that it is usable.**\nThere are techniques that allow us to convert temporal complexity into spatial complexity, such as dynamic programming, which we will see in the following article.
