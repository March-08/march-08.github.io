Apply best practices and learn to use GitHub Actions to build robust code\n![](https://towardsdatascience.com/wp-content/uploads/2023/12/0uDduypwPIxlgtSkT-scaled.jpg)\n<empty-block/>\n### Introduction\nDealing with **integrations, deployment, scalability** and all those topics that make Machine Learning projects a real product is a job on its own. There is a reason why there exist different job positions ranging from data scientist to ML Engineer and MLOps. Still, even if you don’t need to be an expert on these topics, it is **good to have some standard well-defined practices that can help you when you kick off a project**. Certainly! In this article, I outline the best practices I’ve developed – a balance between code quality and the time invested in implementing them. I run my code on [Deepnote](https://deepnote.com/), which is a cloud-based notebook that’s great for collaborative [Data Science](https://towardsdatascience.com/tag/data-science/) projects.\n<empty-block/>\n### Start Simple – Readme\nThis may seem trivial but **try to keep a Readme file more or less up to date**. If it costs you little time, and you like it, also try to make a Readme that looks good. Include imagine headers icons or whatever. This file must **be clear and understandable**. Remember that in a real project, you will not only be working with other developers but also with salespeople, and project managers, and every now and then they might have to read the Readme to understand what you are working on.\nYou can find [here ](https://github.com/othneildrew/Best-README-Template)a really nice readme template!\n> [GitHub – othneildrew/Best-README-Template: An awesome README template to jumpstart your projects!](https://github.com/othneildrew/Best-README-Template)\n### Use virtual environments, your laptop will be happy\nYou probably know this better than I do, in order to develop a cool project we need external libraries. Often a lot of them! These **libraries may have dependencies or conflicts**. That is why it is a **good idea to create virtual environments**. A virtual environment helps you **to have projects isolated from each other**, to have completely different development environments. Usually, to do this in Python you **use pip or conda**.\n> pip\nI personally am a fan of pip. Here is how to create and activate a virtual environment.\n```plain text\n#create virtual env\npython3 -m venv .venv\n```\n```plain text\n#activate virtual env\nsource .venv/bin/activate\n```\nNow you’re allowed to install all the libraries you wish!\n### Create a Requirements file, your colleagues will be happy\nThere is **no point in writing code**, especially in a field like [Machine Learning](https://towardsdatascience.com/tag/machine-learning/), if **you do not allow reproducibility** of the code and experiments. Surely the **place to start is to create a requirements.txt** file\n> [Requirements File Format – pip documentation v23.3.2](https://pip.pypa.io/en/stable/reference/requirements-file-format/)\nI **can’t run code written by someone else if I don’t know what libraries that person has installed** to run the code. For this reason, you should keep a text file named ***requirements.txt*** in which you **enter the names of all the libraries**. You can edit this file manually, that is, every time you install a library with pip, you also enter the name of the library in the requirements. Or you can use a **useful pip command, to automatically enter all the libraries installed** in your virtual environment directly into the requirements. Let’s see how to do this.\nIf you run the following command\n```plain text\npip freeze\n```\nyou will see a list of all installed libraries appear in the terminal. Now just use a terminal trick to redirect the output of this command to the requirements.txt file instead of the terminal display.\n```plain text\npip freeze > requirements.txt\n```\nIf you check your requirements now you will see that they have updated automatically!\nIf you want to automatically install all requirements in a new virtual environment you can run the following:\n```plain text\npip intall -r requirements.txt\n```\n### Format your code with Black\nMany of the libraries I use in this article do much more than what I describe. But as I anticipated my purpose is solely to have some sort of routine to follow when I develop.\n> [black](https://pypi.org/project/black/)\n**I use Black to format code clearly** and neatly. Here is the command you can use to run black:\n```plain text\nfind src -name '*.py' -exec black {} +\n```\nIn the command, we specified to edit all [Python](https://towardsdatascience.com/tag/python/) files (\\*.py) within the src directory.\n### Analyse your code with PyLint\nPyLint is another extremely useful library that I suggest you start using.\n> [pylint](https://pypi.org/project/pylint/)\n**PyLint automatically checks for errors in your code, forces the use of standards, and also checks for code smells**, such as imports that have never been used. PyLint also assigns a score from 1 to 10 to the quality of your code.\n```plain text\npylint --disable=R,C src/*.py\n```\nYou will notice that I have modified the command, disabling two flags (R, and C). In this way, PyLint will not issue warnings or alerts for problems related to refactoring and conventions.\nThe output should look like this:\n![](https://towardsdatascience.com/wp-content/uploads/2023/12/1RXx1vTvwsJgex3tSIgltMA.png)\n<empty-block/>\n### Run Tests, make sure your code is working\n**How do you know your code always works if you don’t use tests?** Get in the **habit of creating simple unit tests**, which you can always extend when you write some function. A unit test is nothing more than another function that will pass a sequence of inputs to the function you want to test and see if the output is as expected.\nYou can implement unit tests in a variety of ways; a widely used library in Python is [PyTest](https://docs.pytest.org/en/7.4.x/).\n> pytest: helps you write better programs – pytest documentation\nI usually create a *src* sister folder, called *test*, in which I collect all my unit tests.\nTo launch all the automatically written tests we run the following\n```plain text\npython -m pytest -vv --cov=test\n```\n<empty-block/>\n### I am lazy, I’ll use a Makefile\nAt this point, we have seen many files and many commands launch it. I would say that as a routine it is a bit heavy. I would like something simpler, I don’t have a very good memory😅\nWell, then we can create a **Makefile, which is a file in which we write some instructions**, and **this will launch the commands previously seen for us**. In the Makefile I want to write instructions to **install requirements, format code with black, check code smells with PyLint, and launch tests with PyTest**. Here then is what our Makefile will look like:\n```plain text\ninstall:\n #install\n pip install --upgrade pip&amp;&amp;\n  pip install -r requirements.txt\nformat:\n #format\n find src -name '*.py' -exec black {} +\nlint:\n #lint\n pylint --disable=R,C src/*.py\ntest:\n #test\n python -m pytest -vv --cov=test\n```\nIn this way every time we use this command from the terminal:\n```plain text\nmake test\n```\nIt will actually be run\n```plain text\npython -m pytest -vv --cov=test\n```\nAnd the same thing applies to all the other commands of course in the same way.\nNow our repo already looks much more professional!\n<empty-block/>\n### **Run Everything at every push with GitHub Actions**\nI hope everything is clear so far. At this point whenever we want to make changes to our code before we commit and push to GitHub we’ll run the following commands to make sure everything works as smoothly as possible: *make install, make format, make lint, make test.*\nBut developers like to automate everything. So, **isn’t there a way that automatically runs this entire process on every git push?** Well yes, there is, just make use of **GitHub Actions**!\nWith GitHub actions, we can **set triggers**, that is, specify events that will trigger commands, and the commands in our case are all in the Makefile.\nTo create GitHub actions, we need to **create the .github/workflows folder** in our working directory. Within this new directory, we **create a YML file** that we can for example name **mlops.yml**.\nIn this file, we can specify several things. First of all, any name we want.\nAfter that, we specify the event (or events) that will trigger the commands, in this case \\[push\\]. Then, we deal with the *steps* (of which the first part I don’t even remember exactly what is, but fortunately there is Google and ChatGPT from which we can copy and paste 😅 ).\n```plain text\nname: Python application with Github Actions\n\non: [push]\njobs:\n  build:\n    runs-on: ubuntu-latest\n\n    steps:\n      - uses: actions/checkout@v4\n      - uses: actions/setup-python@v4\n        with:\n          python-version: \"3.10\"\n\n      - name: Install dependencies\n        run: |\n          make install\n\n      - name: Lint with pyLint\n        run: |\n          make lint\n\n      - name: Test with pytest\n        run: |\n          make test\n\n      - name: Format code\n        run: |\n          make format\n\n      - name: Build container\n        run: |\n          make build\n```\nThis should be the code structure of the entire project\n```plain text\n-.github/workflows\n  - mlops.yml\n-.venv\n-src\n-test\n-requirements.txt\n-Makefile\n-Readme.md\n```\nThat’s it!\nNow you should be able to check on GitHub all the commands running at every push on the *actions* tab.\n![](https://towardsdatascience.com/wp-content/uploads/2023/12/1DXh6vRNglx99lAYmOq0mNg.png)\n<empty-block/>\n## **Final Thoughts**\nIn this article, I have shown you some best practices that I use when developing projects in Python to ensure some quality in the code and protect my GitHub repository from not very functional commits.
