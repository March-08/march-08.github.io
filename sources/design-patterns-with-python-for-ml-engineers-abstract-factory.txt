**Introduction**\nA pattern describes a frequently recurring problem and proposes a possible solution in terms of class/object organization that generally found to be effective in solving the problem itself.\nDesign Patterns are characterized by four main elements:\n- **Name: **mnemonic reference that allows us to identify the problem and solution in one or two words.\n- **Problem: **description of the problem and context to which the pattern is intended to provide a solution.\n- **Solution: **describes the basic elements that constitute thesolution and the relationships between them.\n- **Consequences: **specifies the possible consequences that the application of the proposed solution may entail.\nThere are several categories of design patterns but two main criteria have been identified:\n**What the pattern refers to:**\n- **Objects**: relationships between objects that can change atexecution.\n- **Classes**: concern relationships between classes and subclasses.\n**What the pattern does (purpose) :**\n- **Creational**: concerns the process of creating objects.\n- **Structural**: concerns the composition of classes and objects.\n- **Behavioural**: defines how classes and objects interact anddistribute responsibilities among themselves.\nBut how many design patterns exist? In the following figure, you will see a list of design patterns structured in a table according to their scope and purpose. In the following articles, we would go through the most common design patterns.\n![](https://miro.medium.com/v2/resize:fit:700/1*uZukZlYEh8XQHTByAchAkw.png)\nDesign Patterns (Image by Author)\n## **How to define a design pattern?**\nA design pattern is defined by some fundamental properties that describe and facilitate its use. These properties are :\n- **Structure**: graphical representation via UML of the classes involved andof their relationships. (1) **Participants**: classes participating in the pattern, their relationships and responsibilities. (2) **Collaborations**: how the various classes collaborate to achieve the objectives.\n- **Consequences**: advantages and disadvantages of using the pattern and possible side effects of using the pattern.\n- **Implementation**: techniques and suggestions for implementationwith reference also to specific programming languages.\n- **Example source code**: code fragments that provide aguide for implementation.\n- **Known uses**: examples of use in existing systems.\n- **Related patterns**: differences and most important relationships with other patterns.\n# **Abstract Factory**\nThe easiest way to understand how to define a design pattern is in my opinion show an example, let’s start with the Abstract Factory.\nWe often find ourselves having to create new objects that are similar but different from each other. Suppose we have to create several cars, all of them will share several properties such as having four wheels and a steering wheel, and they will also share several functionality such as *accelerate *and *stop (break)*.\nBut if we want to create cars that have their own peculiarities such as *“vintage cars” *and *“racing cars*” these will have attributes and functions that are particular to the subclass.\nAnd this is precisely where the design pattern comes to our aid, to help us easily and dynamically manage the creation of these objects that share properties and functions but each is different from another. The first thing the Abstract Factory recommends is to explicitly declare interfaces (classes that cannot be instantiated) for each product, for example for a *car* and a *bike *product. Then you can have all product variants implement the interfaces so we are sure they will share attributes and functions that are common to all objects , such as *accelerate() *and *stop()*.\n![](https://miro.medium.com/v2/resize:fit:700/1*5A1u9DYlGqp50X8jmymxFg.png)\nImage by Author\nNow we define an interface of an abstract factory that is, an interface of a class with functions that allow us to create a new object of the type *Car *or *Bike*. The factory that will actually create the car of the type *race *or *vintage *will have to implement the methods of the interface.\n![](https://miro.medium.com/v2/resize:fit:700/1*6h7aWb4lw-HO0flAFi1rDw.png)\nImage by Author\nThis way you will have a factory for each specific subproduct, and you will not have to add attributes and functions during the creation of each individual object.\nIn addition, a customer who uses a factory specification such as RaceFactory will get a race car. After a few months, when he goes to buy a new bike, it will also be a race bike because it comes from the same factory, so he won’t have to worry about having products with different styles.\n![](https://miro.medium.com/v2/resize:fit:700/1*r96jfCD-dqp4S-3jAW5xbQ.png)\nAbstract Factory UML (Image by Author)\n## **Let’s code**\nIf so far you are still unclear about what this design pattern is for and how it works, welcome to the club! If you are like me, programming will clear your mind.\nFirst, we create an AbstractFactory, in Python, an abstract class is a class that inherits ABC, as in the following example. This abstract class has two methods, one to create a car and one to create a bike. So any class that inherits this abstract class will have to implement these two methods in its own way.\n```python\nfrom __future__ import annotations\nfrom abc import ABC, abstractmethod\n\n\nclass AbstractFactory(ABC):\n  \n  @abstractmethod\n  def createCar(self) -> Car:\n      pass\n\n  @abstractmethod\n  def createBike(self) -> Bike:\n      pass\n\n\nclass RaceFactory(AbstractFactory):\n  \n  def createCar(self) -> Car:\n      return RaceCar()\n\n  def createBike(self) -> Bike:\n      return RaceBike()\n\n\nclass VintageFactory(AbstractFactory):\n  def createCar(self) -> Car:\n    return VintageCar()\n\n  def createBike(self) -> Bike:\n      return VintageBike()\n```\n<empty-block/>\nIn both RaceFactory and VintageFactory we have methods that create a car and a bike. You see that both return an object of type Car and Bike (indicated by the symbol: ***-\\> Car***).But actually, Car and Bike are abstract classes the actual classes will be RaceCar and VintageCar (RaceBik and VintageBike). In this way, each Factory will build the cars (or bikes) that relate to it.In this case, the only thing that changes between a RaceCar (RaceBike) and VintageCar (VintagBike) is that it prints *“fast”* instead of *“slowly”*.\n```python\nclass Car(ABC):\n\n  @abstractmethod\n  def accelerate(self) -> str:\n    pass\n    \n  @abstractmethod\n  def stop(self) -> str:\n    pass\n\n\nclass RaceCar(Car):\n  \n  def accelerate(self) -> str:\n      return \"Accelerate really fast!\"\n\n  def stop(self) -> str:\n      return \"Stop really fast!\"\n\nclass VintageCar(Car):\n  \n  def accelerate(self) -> str:\n      return \"Accelerate really slowly...!\"\n\n  def stop(self) -> str:\n      return \"Stop really slowly...\"\n```\n```python\nclass Bike(ABC):\n  \n  @abstractmethod\n  def turn(self) -> str:\n      pass\n\n  @abstractmethod\n  def honk(self) -> str:\n      pass\n\n\nclass RaceBike(Bike):\n  def turn(self) -> str:\n      return \"Turn really fast!\"\n\n  def honk(self) -> str:\n      return \"Honk loudly!\"\n\n\nclass VintageBike(Bike):\n\n  def turn(self) -> str:\n      return \"Turn really slowly...\"\n\n  def honk(self) -> str:\n      return \"Honk quietly...\"\n```\nNow the customer will choose one of the Factories (taking it as input) and then he will create a car and a bike , and he will be sure that the car and the bike will have the same style, they will both be either race or vintage type.\nAfter creating them he will test them using the methods that the Car and Bike classes provide.\n```python\ndef client_code(factory: AbstractFactory) -> None:\n  \n  car = factory.createCar()\n  bike = factory.createBike()\n\n  print(f\"{bike.turn()}\")\n  print(f\"{bike.honk()}\", end=\"\")\n```\n```python\nif __name__ == \"__main__\":\n \n  print(\"Client: checks product style\")\n  client_code(RaceFactory())\n\n  print(\"\\n\")\n\n  print(\"Client: checks product style\")\n  client_code(VintageFactory())\n```\n#
