Introduction to ERC-8004: The Trust Layer for AI Agents\nERC-8004 creates a public discovery and trust layer for autonomous AI agents through three lightweight on-chain registries: Identity, Reputation, and Validation.\n## **Introduction**\nWe're entering the age of AI agents, autonomous software that doesn't just answer questions, but coordinates with other software to deliver real-world outcomes.\nImagine a travel agent that syncs with a budgeting agent and a carbon-impact agent to plan your trip. Or factory agents that negotiate supply, price, and routing across continents. Or health agents that assemble second opinions within minutes.\nBut there's a problem: how do agents from different organizations trust each other?\nWhen a human hires a contractor, they check references, read reviews, and verify credentials. AI agents need the same capabilities, but at machine speed and without human intermediaries. This is the \"trust gap\" that ERC-8004 is designed to solve.\n### **What you will learn:**\nBy the end of this article, you will:\n- Understand what ERC-8004 is and the problem it solves\n- Know how the three core registries (Identity, Reputation, Validation) work together\n- Have a clear path to start building with ERC-8004\nThis article is the first of a series of guides to help you understand and build using the new ERC-8004 standard.\n---\n### **What problems does ERC-8004 Solve?**\nToday's agent communication protocols like [**Google's Agent-to-Agent (A2A)**](https://a2a-protocol.org/latest/) and the [**Model Context Protocol (MCP)**](https://modelcontextprotocol.io/docs/getting-started/intro) handle how agents talk to each other. But they don't address three critical questions:\n- **Discovery**: How do agents find each other across organizational boundaries?\n- **Identity**: How do agents know who they're dealing with?\n- **Trust**: How do agents evaluate whether another agent is reliable?\nWithout answers to these questions, the agent economy fragments into walled gardens controlled by large platforms, or becomes a playground for fraud.\n---\n## **What is ERC-8004?**\n[**ERC-8004**](https://eips.ethereum.org/EIPS/eip-8004), officially titled \"Trustless Agents\", is an Ethereum standard that creates a public discovery and trust layer for autonomous AI agents. It was proposed in August 2025 by a cross-industry team including contributors from MetaMask, the Ethereum Foundation, Google, and Coinbase.\nAt its core, ERC-8004 extends existing agent communication protocols (like A2A and MCP) with three lightweight on-chain registries:\n1. **Identity registry:** Gives every agent a portable, verifiable identity\n2. **Reputation registry:** Records feedback from clients who have worked with an agent\n3. **Validation registry:** Enables third-party verification of an agent's work\nERC-8004 deliberately keeps things minimal. It doesn't try to solve everything, it provides just enough on-chain infrastructure for trust, while leaving application-specific logic off-chain. This design philosophy means:\n- The standard is lightweight and inexpensive to use\n- Developers can plug in their own trust models\n- The ecosystem can evolve without breaking the core protocol\n### **How ERC-8004 relates to other standards**\nERC-8004 doesn't replace existing protocols, it complements them:\n<table header-row=\"true\">\n<tr>\n<td>**Protocol**</td>\n<td>**What It Does**</td>\n<td>**ERC-8004's Role**</td>\n</tr>\n<tr>\n<td>**A2A** (Agent-to-Agent)</td>\n<td>Agent authentication, messaging, task orchestration</td>\n<td>Adds discovery and trust layer</td>\n</tr>\n<tr>\n<td>**MCP** (Model Context Protocol)</td>\n<td>Servers list capabilities (tools, prompts, resources)</td>\n<td>Agents can advertise MCP endpoints in their registration</td>\n</tr>\n<tr>\n<td>**x402**</td>\n<td>HTTP-native micropayments between agents</td>\n<td>Agents can pay each other on-chain via HTTP and JSON payloads</td>\n</tr>\n<tr>\n<td></td>\n<td></td>\n<td></td>\n</tr>\n</table>\n---\n## **Why ERC-8004 matters**\nAs AI agents proliferate, dozens of projects are creating their own discovery and trust systems. Without a shared standard, the ecosystem risks fragmentation, agents from one platform can't interact with agents from another.\nThis is the coordination problem ERC-8004 addresses. Just as ERC-20 unified the token economy and ERC-721 enabled the NFT market, ERC-8004 aims to create a common foundation for the agent economy.\n### **Why Ethereum?**\nYou might ask: why build this on Ethereum instead of a centralized platform?\nThe answer comes down to neutrality and permanence:\n- **Credibly neutral**: No single company controls who can register or what reputation data gets recorded\n- **Censorship-resistant**: Agent identities and reputation histories can't be quietly deleted or modified\n- **Globally accessible**: Any agent, anywhere, can plug into the same trust infrastructure\n- **Composable**: Smart contracts can read reputation data and make automated decisions\nEthereum isn't where models run, it's where their trust runs. A discovery and accountability backbone that any agent can plug into.\n---\n## **How does ERC-8004 work?**\nERC-8004 introduces three registries that work together. Let's explore each one.\n![](https://ai.ethereum.foundation/blog/images/erc-8004-registries-cycle.jpeg)\n### **1. Identity Registry**\nBefore ERC-8004, there was no standard way to discover agents across organizations. The Identity Registry creates a global, crawlable directory of agents—enabling the equivalent of \"web search\" for AI agents.\nThe Identity Registry gives every agent a unique, portable identity on-chain. It's built on [**ERC-721**](https://ethereum.org/developers/docs/standards/tokens/erc-721/), which means:\n- Each agent identity is an NFT that can be viewed in any wallet or NFT marketplace\n- Ownership can be transferred or delegated\n- The identity is globally unique and censorship-resistant\n**How it works:**\nWhen an agent registers, it receives an **`agentId`** (the NFT token ID). This ID links to a registration file (hosted on chain as Base64, IPFS, or any URL) that contains:\n- Agent name and description\n- Supported endpoints (A2A, MCP, etc.)\n- Wallet addresses on various chains\n- Which trust models the agent supports\n- Support for OASF (Open Agent Service Framework) endpoints\nThink of it as a machine-readable \"passport\" for AI agents.\n### **2. Reputation Registry**\nReputation is collateral for AI agents. Agents with transparent track records can command higher prices and access more opportunities. The Reputation Registry makes this reputation portable, it follows the agent regardless of which platform or application it uses.\nThe Reputation Registry creates an on-chain audit trail of client feedback. When you hire an agent to perform a task, you can leave a rating that becomes part of the agent's permanent record.\n**How it works:**\n1. When an agent accepts a task, it signs a **`feedbackAuth`** authorizing the client to leave feedback\n2. After the task, the client submits feedback: a score (0-100), optional tags, and a link to detailed off-chain data\n3. The feedback is recorded on-chain and linked to the agent's identity\n4. Anyone can query an agent's reputation history\n**Key design decisions:**\n- **Pre-authorization prevents spam**: Only clients the agent has actually worked with can leave feedback\n- **On-chain + off-chain hybrid**: Basic scores are stored on-chain for composability; detailed feedback lives off-chain for cost efficiency\n- **Revocable feedback**: Clients can revoke feedback if disputes are resolved\n- **Response mechanism**: Agents (or anyone) can append responses to feedback\n### **3. Validation registry**\nDifferent tasks require different levels of assurance. Ordering pizza? Reputation feedback is probably enough. Managing a DeFi portfolio or diagnosing medical conditions? You want cryptographic or economic guarantees. The Validation Registry provides a unified interface for all these trust models.\nFor high-stakes tasks, reputation alone isn't enough. The Validation Registry enables independent third-party verification of an agent's work.\n**How it works:**\n1. An agent requests validation by specifying a validator contract and providing a hash of the task data\n2. The validator performs the check (using various methods)\n3. The validator records a response (0-100) on-chain\n**Validation methods:**\nERC-8004 supports multiple validation approaches:\n- **Crypto-economic validation**: Stakers re-run the job and risk their stake on the outcome\n- **Zero-knowledge proofs (zkML)**: Cryptographic proof that computation was performed correctly\n- **TEE attestations**: Trusted Execution Environment oracles verify the work\n- **Trusted judges**: Human or algorithmic arbiters for subjective tasks\n### **How the 3 registries work together**\nHere's how the registries combine in practice:\n1. A client searches the **Identity registry** to discover agents with the right capabilities\n2. The client checks the **Reputation registry** to filter for agents with strong track records\n3. For the chosen agent, the client may check the **Validation registry** to see if the agent's work has been independently verified\n4. After completing the task, the client adds feedback to the **Reputation registry**, building the agent's reputation over time\nThis creates a virtuous cycle: good agents build reputation, attract more clients, and can afford validation, which further increases trust.\n---\n## **ERC-8004 use cases**\nSo what can you actually build with this? Here's where the ERC-8004 standard is gaining traction and possible use cases:\n**DeFi and trading**\n- **Research-as-a-Service**: Portfolio agents discover and hire analysis agents based on reputation scores, then validate high-stakes recommendations via stake-secured re-execution\n- **On-chain credit**: Lending protocols check borrower agent reputation before extending credit\n**Enterprise coordination**\n- **Supply Chain**: Logistics agents from different companies discover each other, verify capabilities via TEE attestation, and build cross-organizational trust without legal agreements\n- **Procurement**: Purchasing agents evaluate vendor agents based on delivery history and payment-verified feedback\n**AI Agent marketplaces**\n- **Specialized services**: Agents offering narrow capabilities (code review, fact-checking, translation) build portable reputation that commands premium pricing\n- **Quality filtering**: Clients filter service providers by feedback from verified transactions (using x402 payment proofs to distinguish real clients from Sybil attacks)\n**Governance & DAOs**\n- **Verified Advisory**: DAOs hire analysis agents for treasury proposals, requiring zkML validation to prove financial models ran correctly\n- **Delegate Reputation**: Governance agents build track records that follow them across protocols\n---\n## **What's **Conclusion**\nERC-8004 represents a foundational step toward an open, permissionless agent economy. By providing standardized registries for identity, reputation, and validation, it enables AI agents to discover each other and build trust, without relying on centralized intermediaries.\n**Key takeaways:**\n- ERC-8004 solves the \"trust gap\" for AI agents operating across organizational boundaries\n- It introduces three lightweight registries: Identity (who are you?), Reputation (are you reliable?), and Validation (can you prove it?)\n- The standard is minimal by design, enabling pluggable trust models and ecosystem evolution\n- Ethereum serves as the neutral coordination layer—not where agents run, but where their trust runs\nThe agent economy is coming.
