**Introduction**\nRecently I’ve been working on the domain-specific fine-tuning of several LLMs. The first and maybe the most important part of this task is to collect, scrape, and clean textual data to feed the LLM. I noticed that my code was becoming messy with many repetitions, because for every identified source I was writing a script from scratch which had a lot of things in common with other scripts in my codebase. I was not following the “***Don’t repeat yourself***” (DRY) principle at all. This is why I decided to implement the Template Design Pattern and make my code base more elegant and efficient.\n## **The Template Design Pattern**\nI won’t repeat here what a design pattern is and how we classify design patterns based on their functionalities, since I’ve written many articles on the subject. If you are interested in reading my previous articles on this topic I will leave some references at the end.\nIn this article, I will show you an **example related to data processing**. Let’s say that in our project we have to deal with different kinds of data that we want to analyze. Some of these data are financial related that we gathered maybe from an external API. But we also have to analyze some surveys, and at the end provide a report to our boss.\nThese two types of data are really different from one another, they represent different things, are formatted differently and so on. But even in this case, we need to perform similar operations on both of them.\nFor sure we have to clean the data and transform them in a format which is more comfortable for us than analysing and saving our results.\nSo even if each single step is ad hoc per data type, the steps (or the algorithm) are the same. So what we can do is to define a general abstract class that defines these steps at a high level.\nCreating an abstract class in Python is sufficient to inherit from ABC, and use the decorator @abstactmethod on the methods we want overridden in concrete subclasses.\nThis class will have a general method named `process_data` where we define the above-mentioned steps. This method goes into the abstract class because it is something all the subclasses will have in common, so we avoid repeating it every time. The abstract methods instead will be re-implemented in each single subclass.\n```plain text\nfrom abc import ABC, abstractmethod\n\n# The abstract class defines a template method and abstract steps.\nclass DataProcessor(ABC):\n    def process_data(self, data):\n        \"\"\"The template method defining the skeleton of data processing.\"\"\"\n        cleaned_data = self.clean_data(data)\n        transformed_data = self.transform_data(cleaned_data)\n        analyzed_data = self.analyze_data(transformed_data)\n        self.save_results(analyzed_data)\n\n    @abstractmethod\n    def clean_data(self, data):\n        \"\"\"Clean the raw data.\"\"\"\n        pass\n\n    @abstractmethod\n    def transform_data(self, data):\n        \"\"\"Transform the data into a useful format.\"\"\"\n        pass\n\n    @abstractmethod\n    def analyze_data(self, data):\n        \"\"\"Analyze the transformed data.\"\"\"\n        pass\n\n    @abstractmethod\n    def save_results(self, results):\n        \"\"\"Save the analysis results.\"\"\"\n        pass\n```\nNow every time we want to create a subclass that deals with a specific data type, we only need to inherit the template class “DataProcessor” and implement each abstract method. Let’s do the class that handles the financial data.\n```plain text\n# Concrete class for processing financial data\nclass FinancialDataProcessor(DataProcessor):\n    def clean_data(self, data):\n        print(\"Cleaning financial data...\")\n        return [d.strip() for d in data if d.strip()]\n\n    def transform_data(self, data):\n        print(\"Transforming financial data...\")\n        return [float(d) for d in data]\n\n    def analyze_data(self, data):\n        print(\"Analyzing financial data...\")\n        return {\n            \"total\": sum(data),\n            \"average\": sum(data) / len(data),\n            \"max\": max(data),\n            \"min\": min(data)\n        }\n\n    def save_results(self, results):\n        print(\"Saving financial analysis results...\")\n        with open(\"financial_results.txt\", 'w') as file:\n            for key, value in results.items():\n                file.write(f\"{key}: {value}\\n\")\n        print(\"Results saved to financial_results.txt\")\n\n```\nWe can do the same for the survey data😀\n```plain text\n# Concrete class for processing survey data\nclass SurveyDataProcessor(DataProcessor):\n    def clean_data(self, data):\n        print(\"Cleaning survey data...\")\n        return [d for d in data if d.isdigit()]\n\n    def transform_data(self, data):\n        print(\"Transforming survey data...\")\n        return [int(d) for d in data]\n\n    def analyze_data(self, data):\n        print(\"Analyzing survey data...\")\n        from collections import Counter\n        return dict(Counter(data))\n\n    def save_results(self, results):\n        print(\"Saving survey analysis results...\")\n        with open(\"survey_results.txt\", 'w') as file:\n            for key, value in results.items():\n                file.write(f\"{key}: {value}\\n\")\n        print(\"Results saved to survey_results.txt\")\n```\nThat’s pretty much it!\nNow we can run our main.py (I’m using here some mock data):\n```plain text\nprint(\"Processing Financial Data:\")\nfinancial_data = [\"100\", \" 200 \", \"300\", \"\", \"400\"]\nfinancial_processor = FinancialDataProcessor()\nfinancial_processor.process_data(financial_data)\n\nprint(\"\\nProcessing Survey Data:\")\nsurvey_data = [\"5\", \"3\", \"4\", \"5\", \"2\", \"\", \"5\", \"3\"]\nsurvey_processor = SurveyDataProcessor()\nsurvey_processor.process_data(survey_data)\n```\nUsing this pattern we avoided having a lot of conditionals, which usually developers put there to redirect the data into the correct processor. Also, imagine you’re working in a big team, and at a certain point your boss wants you to handle a new type of data. A new developer just by following the first template (the abstract class) knows what he has to do. So having your code structured in this way is way more scalable avoiding technical debts in the future.\nFollow me on [Medium](https://medium.com/@marcellopoliti) if you like this article! 😁\n💼
